- [x] CHK017 - Is the contract's nesting depth rule (2-space indent per level) validated against actual terraform-docs rendering behavior for 3+ nesting levels? - [x] CHK018 - Is "maybe" vs "may be" in the deliberately arbitrary statement explicitly called out as intentional in the contract (not just research.md)? [Clarity, contracts/description-style.md §1] **PASS with note**: Contract templates all use correct "maybe" (one word). Research §3 and plan.md document the rationale. Note: spec FR-004 uses "may be" (two words) — cosmetic inconsistency with the AVM canonical text, but contract is the implementation source of truth.
- [x] CHK019 - Does the contract define the heredoc delimiter name (e.g., `DESCRIPTION` vs `DESC` vs `EOT`) to ensure consistency across all 17 variables? ## Scenario Coverage — Edge Cases & Exception Flows

- [x] CHK033 - Are requirements defined for attributes that exist in the pattern's type definition but have no corresponding AVM source description (pattern-only attributes)? [Edge Case, Spec §Edge Cases §4] **PASS**: Edge Cases §4 covers fallback ("use type constraint and Terraform provider documentation"). Contract §5 covers `use_default_log_analytics` (main pattern-only attr). Contract §4 covers cross-reference keys (pattern-only). CHK003 covered `byo_log_analytics_workspace` (fully pattern-authored).
- [x] CHK034 - Are requirements specified for attributes where the AVM source uses a different type than the pattern (e.g., AVM has `string` but pattern wraps it in `optional(string)`)? [Edge Case, Gap] **PASS**: FR-002 determines `(Required)`/`(Optional)` from the pattern's type definition, not AVM's. The pattern type is authoritative for markers. AVM text is used for the behavioral description only (FR-003). No special handling needed — `optional(string)` = `(Optional)`, `string` = `(Required)`.
- [x] CHK035 - Is the fallback behavior specified for when `.terraform/modules/` is not populated (missing AVM source)? [Edge Case, Spec §Dependencies] **PASS**: Spec Dependencies: ".terraform/modules/ must be populated via terraform init." Quickstart Step 1 includes `terraform init`. This is a prerequisite, not a runtime fallback scenario — if modules aren't populated, you can't start.
- [x] CHK036 - Are requirements defined for handling AVM modules that may have been updated since the research extraction (version drift between research and implementation)? [Edge Case, Gap] **PASS with note**: Research extracted from pinned versions (listed in data-model). Module versions are locked in `main.tf` — they don't change unless explicitly updated. If modules are updated before implementation, re-run `terraform init` + re-extract. This is a timing risk, not a requirements gap. Feature is a one-time rewrite.
- [x] CHK037 - Is the behavior specified when a `validation` block's constraint cannot be meaningfully expressed in the description text (e.g., complex regex patterns)? [Edge Case, Spec §FR-017] **PASS with note**: FR-017 says "mention the constraint in the description so users understand the rule without reading HCL." For complex regex, describe the rule in human-readable terms (e.g., "Must be a valid Azure resource ID" rather than reproducing the regex). No explicit guidance on simplification level, but common sense applies — the goal is user understanding, not regex reproduction.
