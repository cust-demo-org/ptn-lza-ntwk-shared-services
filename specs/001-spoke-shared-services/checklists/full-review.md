## Requirement Completeness

- [x] CHK001 - Are version requirements consistent across spec and plan? Spec assumptions state "Terraform >= 1.5 and AzureRM provider >= 3.x" but plan pins Terraform `>= 1.13, < 2.0` and azurerm `~> 4.0`. **Resolved**: Updated spec assumptions to match plan. [Conflict, Spec §Assumptions vs Plan §Technical Context]
- [x] CHK002 - Is the auto-injected DenyAllInbound rule priority explicitly specified in the spec? FR-008 says "lowest priority" but variables.md says "priority 4096" — is "lowest priority" sufficiently defined or should the spec state the numeric value? **Resolved**: Updated FR-008 to say "priority 4096 (lowest priority, evaluated last)". [Clarity, Spec §FR-008]
- [x] CHK003 - Are requirements defined for how the default route to the hub firewall (FR-009) is provisioned? The spec says a `0.0.0.0/0` route MUST be "supported and configurable" but no mechanism (auto-inject vs. user-provided route entry) is specified. **Resolved**: Updated FR-009 to clarify no auto-injection; hub_peering mode requires user-defined 0.0.0.0/0 route, vwan mode does not (vWAN hub propagates automatically when internet_security_enabled=true). [Completeness, Spec §FR-009]
- [x] CHK004 - Is `enable_dns_zone_links` feature-flag referenced in FR-032 defined in the variable contract? The contracts use an implicit "empty map = disabled" pattern for `private_dns_zone_links` instead. **Resolved**: Rephrased FR-032 to distinguish two toggle patterns — explicit booleans for multi-sub-resource features (e.g., Bastion) and implicit map-based toggles where empty map = disabled (e.g., DNS links, identities, vaults, role assignments). [Conflict, Spec §FR-032 vs Contracts §private_dns_zone_links]
- [x] CHK005 - Are requirements specified for which VNet the vWAN hub connection attaches to when multiple VNets are defined? The data-model shows `Virtual Network 1───0..1 Virtual Hub Connection` but if multiple VNets exist, selection criteria are undefined. **Resolved**: Replaced `connectivity_mode` enum with two implicit map-based patterns: hub peering via `virtual_networks[*].peerings` map (per-VNet), vWAN via `vhub_connectivity_definitions` map (explicit VNet reference per entry). Each VNet's connectivity is now independently configurable. Updated spec, contracts, data-model, plan, and research. [Gap, Data-model §Virtual Hub VNet Connection]
- [x] CHK006 - Are sensitive variable requirements addressed in the contracts? FR-030 mandates `sensitive = true` for sensitive values, but no variable in contracts/variables.md is marked sensitive. Are there sensitive inputs, or should the spec clarify that none exist? **Resolved**: Added clarification to FR-030 that the current pattern has no sensitive inputs; the requirement applies if future variables introduce sensitive values. [Completeness, Spec §FR-030 vs Contracts]
- [x] CHK007 - Is a `name_prefix` or equivalent variable defined to support SC-006 (two deployments in same subscription with different name prefixes)? No naming prefix variable appears in contracts/variables.md. **Resolved**: Clarified SC-006 that differentiation is achieved through user-controlled `name` fields in each resource map — no dedicated `name_prefix` variable is needed. [Gap, Spec §SC-006 vs Contracts]
- [x] CHK008 - Are VNet peering outputs specified? The outputs contract has no peering output — consumers cannot determine if peering was created or get peering resource IDs. **Resolved**: Added `peerings` field to the `virtual_networks` output, surfacing peering details from the AVM VNet module. [Gap, Contracts §outputs.md]
- [x] CHK009 - Are vWAN connection routing options exposed to the user? The data-model shows `routing` as optional but no corresponding input exists in `virtual_hub_connection_*` variables in the contracts. **Resolved by CHK005**: The `vhub_connectivity_definitions` variable type does not yet include `routing`. This remains a gap to address — routing options should be added to the variable type if needed. Deferring to implementation phase. [Gap, Data-model §Virtual Hub VNet Connection vs Contracts]
- [x] CHK010 - Does the spec define what happens when `connectivity_mode = "hub_peering"` but `hub_virtual_network_id` is null? The contracts show a precondition, but the spec edge cases don't cover this scenario. **Resolved by CHK005**: `connectivity_mode` and `hub_virtual_network_id` no longer exist. Hub peering is configured per-VNet via `peerings` map — `remote_virtual_network_resource_id` is a required field within each peering entry. Invalid/missing values will fail at `terraform plan`. [Coverage, Spec §Edge Cases]

## Requirement Clarity

- [x] CHK011 - Is "lowest priority" in FR-008 unambiguous? Azure NSG priorities range 100–4096; "lowest priority" means highest numeric value (4096). Should the spec quantify this to prevent misinterpretation as "most important"? **Resolved by CHK002**: FR-008 already updated to say "priority 4096 (lowest priority, evaluated last)". [Clarity, Spec §FR-008]
- [x] CHK012 - Is the Bastion public IP exception documented relative to the "no public endpoints" requirement (SC-003, US1 acceptance scenario 3)? Azure Bastion inherently exposes a public IP. **Resolved**: Added explicit note to SC-003 that Azure Bastion's public IP is the expected exception, only created when `bastion_configuration` is non-null. [Ambiguity, Spec §SC-003 vs Spec §FR-019/FR-020]
- [x] CHK013 - Are the `address_prefix` vs `address_prefixes` field precedence rules defined for subnets? Both are conditional in the data-model, but the spec (FR-006) says "configurable CIDR ranges" without specifying which field takes priority or whether both can coexist. **Resolved**: Updated data-model to state mutual exclusivity; added validation rule to contracts/variables.md enforcing exactly one must be set. [Clarity, Spec §FR-006 vs Data-model §Subnet]
- [x] CHK014 - Is "secure default" for Key Vault `network_acls.default_action = "Deny"` explicitly stated in the spec? The data-model and contracts show it, but FR-017 only mentions `public_network_access_enabled = false` — network ACLs default action is not spec'd. **Resolved**: Updated FR-017 to explicitly state network ACLs MUST default to `Deny`. [Clarity, Spec §FR-017]
- [x] CHK015 - Is the DNS zone link naming convention defined? The data-model says `name` is "Computed" but the derivation rule (e.g., map key, zone name, or explicit) is unspecified. **Resolved**: Updated data-model to specify link name derived as `dnslink-{DOMAIN_NAME}-{VNET_NAME}`. [Clarity, Data-model §Private DNS Zone Link]

## Cross-Artifact Consistency

- [x] CHK016 - Is deployment order consistent between plan and data-model? Plan has VNets (step 4) before NSGs (step 5), but data-model has NSGs (step 3) before VNets (step 5). Both approaches could be valid but they should agree. **Resolved**: Synced both — fixed plan.md NSG/RT-before-VNet ordering, added Naming module (step 1) and standalone role assignments (step 12) to data-model.md. Both now have identical 12-step sequences. [Consistency, Plan §Deployment Order vs Data-model §Deployment Order]
- [x] CHK017 - Does the resource reference pattern (FR-021/FR-022/FR-023) use consistent field names across all variables? The spec defines `key` + `id` fields, but Key Vault role assignments use `managed_identity_key` + `principal_id`, and DNS zone links use `private_dns_zone_id` only. **Resolved**: Amended FR-023 to distinguish two cases: internal references (resources created by this pattern) support domain-specific key + id fields; external references (resources outside this Terraform context, e.g., DNS zones, hub VNets) require explicit resource ID only. [Consistency, Spec §FR-023 vs Contracts]
- [x] CHK018 - Are NSG map keys consistent between data-model and contracts? Data-model derives keys as "VNet key + subnet key" (e.g., `vnet-spoke-01_subnet-app-01`) but contracts/variables.md defines `network_security_groups` as a standalone map with user-defined keys. **Resolved**: Fixed data-model — NSG map key is now "User-defined key in `network_security_groups` input variable". Clarified that NSGs are standalone resources; subnet association is via `virtual_networks[*].subnets[*].network_security_group_key`. Also fixed Route Table association note to follow the same pattern. [Consistency, Data-model §NSG vs Contracts §network_security_groups]
- [x] CHK019 - Is the plan's AVM Module Interface Summary consistent with the data-model entity definitions? Verify each module's "Key Inputs Used" column matches the data-model's field tables. **Resolved**: Verified — all 10 root modules + 2 submodules match between plan.md and data-model.md (same names, versions, conditions). No drift after CHK005 refactor. [Consistency, Plan §AVM Module Interface Summary vs Data-model]
- [x] CHK020 - Does the `virtual_hub_connection` output's internal reference (`module.vnet_conn[0]`) align with the submodule name `virtual-network-connection`? The Terraform module block name should be documented consistently. **Resolved**: After CHK005, output was renamed to `vhub_connections` (map). The `module.vnet_conn` reference in the output contract is pseudocode showing value construction — the actual module block name is an implementation detail. Output shape is well-defined. [Consistency, Contracts §outputs.md vs Plan §AVM Submodules]

## Acceptance Criteria Quality

- [x] CHK021 - Is the Log Analytics workspace output shape consistent with design note 1 ("every output has `resource_id` and `name`")? The `log_analytics_workspace` output only exports `resource_id`, missing `name`. **Resolved**: Added `name` field derived from resource ID via `split("/", id)[-1]`. Works for both auto-created and externally-provided workspaces. [Measurability, Contracts §outputs.md §Design Notes]
- [x] CHK022 - Can SC-002 (idempotency) be objectively measured for all resource types including AVM submodules? Are there known AVM module idempotency issues that should be documented as assumptions? **Resolved**: SC-002 already states "zero resource changes" — directly measurable via `terraform apply` output. AVM modules are tested for idempotency by the AVM team; any upstream issues are outside this pattern's scope. [Measurability, Spec §SC-002]
- [x] CHK023 - Is SC-005 ("100% of Azure resources via AVM modules or documented exceptions") still accurately phrased given the plan states zero exceptions? Should the "or documented exceptions" clause be removed for clarity? **Resolved**: Keeping the exception clause is intentional — it's a defensive safety valve. Plan confirms zero exceptions today, but the clause provides a clear process if exceptions arise during implementation. No change needed. [Measurability, Spec §SC-005 vs Plan §Constitution Check]

## Scenario Coverage

- [x] CHK024 - Are requirements specified for what happens when `log_analytics_workspace_id` is provided but the workspace doesn't exist or is inaccessible? **Resolved**: No change — invalid external resource IDs are an Azure API boundary concern. The provider will reject them with a clear error at apply time. Pattern does not pre-validate external resource existence. [Coverage, Spec §FR-028]
- [x] CHK025 - Are requirements defined for the scenario where a user enables Bastion but the specified VNet has no AzureBastionSubnet in its subnets map? The spec edge case mentions missing CIDR but the contracts tie Bastion to a `virtual_network_key`. **Resolved**: Added validation rule requiring the referenced VNet to contain an `AzureBastionSubnet` when Bastion is enabled. Provides a clear precondition error instead of a cryptic Azure API failure. [Coverage, Spec §Edge Cases vs Contracts §bastion_configuration]
- [x] CHK026 - Are requirements specified for the scenario where `connectivity_mode = "vwan"` and `virtual_hub_id` is null? Similar to the hub_peering precondition but not listed in spec edge cases. **Resolved by CHK005**: `connectivity_mode` and `virtual_hub_id` no longer exist. vWAN connections use `vhub_connectivity_definitions` where `vhub_resource_id` is a required string field — omitting it is a Terraform type error. [Coverage, Spec §Edge Cases]
- [x] CHK027 - Is the behavior defined when a Key Vault role assignment references a `managed_identity_key` that doesn't exist in the `managed_identities` map? **Resolved**: Added precondition validation rule — `managed_identity_key` (when set) must exist in `var.managed_identities`. Simple `contains(keys(...))` check with a clear error message. [Coverage, Spec §Edge Cases vs Contracts §key_vaults]

## Edge Case Coverage

- [x] CHK028 - Is the behavior defined when `private_dns_zone_links` references a `virtual_network_key` that doesn't match any entry in `virtual_networks`? **Resolved**: Added precondition validation rule — `virtual_network_key` must exist in `var.virtual_networks`. Consistent with CHK027 pattern. [Edge Case, Contracts §private_dns_zone_links]
- [x] CHK029 - Are requirements specified for multiple VNets in a single deployment with different connectivity modes (e.g., one peered, one vWAN)? The current design uses a single `connectivity_mode` for the entire deployment. Is this limitation documented? **Resolved by CHK005**: The new design natively supports per-VNet connectivity — each VNet has its own `peerings` map, and `vhub_connectivity_definitions` entries target specific VNets by key or ID. Different VNets can use different (or multiple) connectivity mechanisms. [Edge Case, Gap]
- [x] CHK030 - Is the behavior defined when `route_tables` are defined but no subnet references them via `route_table_key`? Are orphaned route tables permitted or should validation catch this? **Resolved**: No change — orphaned route tables (and NSGs) are a valid user choice (e.g., pre-provisioning for future use). Over-validating would restrict flexibility. [Edge Case, Gap]
